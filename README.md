# Дипломная работа слушателя TV: Облачное хранилище (Cloud Service)

REST-сервис, предоставляющий интерфейс для загрузки файлов, вывода списка уже загруженных файлов пользователя, их переименования, скачивания и удаления. Проект разработан в рамках дипломной работы профессии "Java-разработчик".

## Технологии и инструменты
* **Backend:** Java 21, Spring Boot (Web, Data JPA, Security)
* **Безопасность:** JWT (JSON Web Token)
* **База данных:** PostgreSQL
* **Инфраструктура:** Docker, Docker Compose
* **Тестирование:** JUnit 5, Mockito (Unit-тесты), Testcontainers (Интеграционные тесты)
* **Сборщик пакетов:** Gradle
* **Frontend:** Vue.js, TypeScript

## Архитектура приложения

```
[Frontend (Vue.js)]
│
│ HTTP / REST API (JSON)
│ Header: auth-token
▼
[Backend (Spring Boot)]
│
│ Spring Data JPA (Hibernate)
▼
[Database (PostgreSQL)]
```
Приложение построено по классической трехуровневой архитектуре (Controller - Service - Repository).
* **Controller:** Обрабатывает HTTP-запросы от фронтенда (согласно предоставленному YAML-контракту).
* **Service:** Содержит бизнес-логику валидации, работы с файлами и проверки прав.
* **Repository:** Отвечает за сохранение и извлечение данных из PostgreSQL с использованием Spring Data JPA.

Безопасность реализована в режиме `stateless` (без сессий). При успешной авторизации пользователю выдается JWT токен, который фронтенд передает в заголовке `auth-token` при каждом последующем запросе. Специальный фильтр (`JwtFilter`) перехватывает запросы и авторизует пользователя в контексте Spring Security.

### Структура базы данных
В базе данных `cloud_db` реализованы две связанные таблицы:
1. `users` — хранит учетные записи пользователей.
    * `login` (PK) — логин пользователя (email).
    * `password` — пароль.
2. `files` — хранит метаданные и бинарное содержимое файлов.
    * `id` (PK) — уникальный идентификатор.
    * `filename` — имя файла.
    * `size` — размер файла в байтах.
    * `file_content` (LOB) — массив байтов (содержимое файла).
    * `user_login` (FK) — внешний ключ на таблицу `users`.

## Реализованный функционал (REST API)

Сервис полностью реализует все методы, описанные в YAML-файле задания. Предоставляются следующие эндпоинты:

* `POST /login` — Авторизация пользователя и выдача JWT-токена.
* `POST /logout` — Выход пользователя (удаление/деактивация токена).
* `POST /file` — Добавление (загрузка) файла в облако.
* `DELETE /file` — Удаление файла.
* `GET /file` — Скачивание файла.
* `PUT /file` — Переименование файла.
* `GET /list` — Вывод списка файлов пользователя.

**Ключевые особенности реализации:**
* **Конфигурация:** Все настройки приложения (порт, параметры подключения к БД, секреты для JWT и правила инициализации) вынесены и вычитываются из единого файла настроек `application.yml`.
* **Хранение данных:** Информация о пользователях (логины и пароли для авторизации), а данные хранятся в реляционной базе данных PostgreSQL.

## Инструкция по запуску

### 1. Запуск Backend и Базы данных
Убедитесь, что у вас установлен и запущен **Docker** и **Docker Desktop**.

1. Клонируйте репозиторий бэкенда на свой компьютер.
2. В корне проекта выполните команду для запуска базы данных:
   ```bash
   docker-compose up -d
   ```
3. Запустите приложение (CloudserviceApplication.java) через вашу IDE или с помощью команды Gradle. Бэкенд запустится по умолчанию на порту 8080.

Важно!: В проекте есть готовый скрипт data.sql, при первом запуске он автоматически создаст тестового пользователя (admin@mail.ru / 12345).

### 2. Запуск Frontend

1. Установка Node.js

Убедитесь, что у вас установлен Node.js (версия 19.7.0 или выше). 
Можно использовать NVM для управления версиями. Например мои старые рабочие проекты используют версию 16.16.0, и очень удобно с помощью NVM переключаться между версиями. Скачать можно тут:

https://github.com/coreybutler/nvm-windows/releases

Нужен файл: nvm-setup.exe

Установка версий Node.js происходит через PowerShell командой, например:

```bash
nvm install 16.16.0
```
или
```bash
nvm install 20.12.2
```
Чтобы переключиться между версиями используем в том же PS команду:

```bash
nvm use 16.16.0
```
или
```bash
nvm use 20.12.2
```
2. Перейдите в папку с фронтенд-приложением.
3. Убедитесь, что в файле .env указан корректный URL для связи с бэкендом:
```bash
VUE_APP_BASE_URL=http://localhost:8080
```
4. Установите зависимости.

Простого *npm install* недостаточно. Будет множество ошибок из-за несовместимости библиотек. Нужно использовать флаг - *legacy-peer-deps*
```bash
npm install --legacy-peer-deps
```
5. Запуск сервера

```bash
npm run serve
```
6. Доступ к приложению по умолчанию будет на порту 8081. Если backend запустился по умолчанию ранее на 8080, а затем поднимаем front.

http://localhost:8081

## Тестирование

Проект покрыт автоматизированными тестами:

* Unit-тесты: Изолированное тестирование сервисов (AuthService, FileService) с использованием заглушек (Mockito).

* Интеграционные тесты: Проверка полного контекста приложения в связке с реальной БД PostgreSQL, поднимаемой в Docker через Testcontainers перед выполнением тестов.
